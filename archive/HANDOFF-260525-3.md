## What shipped this session

### PR #69 — restore Home floating island + anchor at OLD Home position (merged)

Commit `8d928b3` on `main`. 11 files, +184 / -192.

Two compounding bugs PR #68 introduced on Home:

**(1) Tab bar height grew by ~34pt in full-edge mode**, which
pushed the buttons (and the mini-player stacked above them) up.
`AtlasTabBar` had a `bottomExtensionHeight` that added the
home-indicator safe-area inset (~34pt) to the painted area below
the buttons. The view's VStack grew taller, and because
`ContentView`'s ZStack is bottom-aligned with
`.ignoresSafeArea(.bottom)`, the extra height shoved everything
above (buttons + mini-player) up. The intent had been to fill the
bottom strip opaquely; doing it via *height* was wrong.

**(2) Preference-driven `moduleGeometry` got stuck at `.fullEdge`**
after popping back from a detail screen. So Home ended up
rendering in full-edge geometry — no floating island, AND because
of (1) at the wrong (higher) position. Owner: *"my home screen
with the floating island was what i wanted. now you messed it up.
it's no longing floating (only the drawer is floating, not the
miniplayer or tab bar), plus it appears you've moved everything
upwards."*

Fix:

**(A) Anchor at the OLD Home position.** `AtlasTabBar.body` is a
fixed-height `VStack(spacing: 0)` in both modes: a 56pt painted
button row + an 8pt outer strip = 64pt total. The buttons sit at
the SAME screen-y across all tabs and pushed details. Only what's
painted in the 8pt strip below the buttons differs:
- Home: transparent (the floating-island gap — map shows through;
  home indicator pill sits over map)
- Everywhere else: opaque `secondaryBackground` (the bar continues
  to the screen bottom as one continuous surface)

The safe-area zone beneath the buttons is already covered by the
painted 56pt button row, since `ContentView`'s ZStack
`.ignoresSafeArea(.bottom)` and the VStack is bottom-aligned. The
56pt painted area's bottom edge sits at screen_bottom - 8 (the
outer strip), and from there UP it covers the entire home-indicator
inset on Face-ID phones. We don't need to inflate the view's
height to "fill" the safe area.

**(B) Deterministic geometry detection.** Replaced the
`AtlasModuleGeometryKey` PreferenceKey + `.atlasModuleGeometry(...)`
modifier with an `@Observable AtlasNavigationState` class.
`ContentView` owns it as `@State` and injects via
`.environment(navState)`. Each pushed view (`TourDetailView`,
`MakerView`, `ManageDownloadsView`, `SearchView`) calls
`navState.push()` on `.onAppear` and `navState.pop()` on
`.onDisappear`. `pushedDepth > 0` ⇒ `isShowingDetail`. ContentView
derives module geometry as:

```swift
private var moduleGeometry: AtlasModuleGeometry {
    if navState.isShowingDetail { return .fullEdge }
    return selectedTab == .home ? .floatingIsland : .fullEdge
}
```

Deterministic — no preference-bubbling-up flakiness, no stuck
values after pop.

**(C) `AtlasBottomModule.height` is now a constant 126pt** across
modes (62pt mini-player view + 56pt painted tab bar + 8pt outer
strip). All callers — `HomeView`'s `floatingIslandHeight`, the
`safeAreaInset(.bottom)` reservations on Library / Settings /
Maker / ManageDownloads / Search, and `TourDetailView`'s
`actionBarHeight` — read the same value. The
`extendsToScreenEdges` parameter stayed (with a default of `false`)
for call-site documentation but doesn't affect the result.

## Process notes / gotchas for next session

- **The PreferenceKey approach for cross-cutting layout state is
  fragile in SwiftUI.** It can get stuck after NavigationStack
  pops (sticky `.fullEdge` after returning to Home from a detail).
  The Observable + push/pop pattern is much more deterministic.
  Worth remembering if a similar "what's currently visible
  controls a parent layout decision" question comes up again —
  reach for Observable, not PreferenceKey.

- **`.ignoresSafeArea(.bottom)` + bottom-aligned ZStack is the
  cleanest way to extend a view through the home-indicator
  zone.** The painted area covers the safe area for free; no need
  to read `UIWindowScene.safeAreaInsets.bottom` and inflate the
  view's height. Doing it via height shoves the rest of the
  vertical stack up. Worth keeping in mind for any future "let
  this surface bleed to the very bottom" requirement.

- **`AtlasBottomModule.height` being constant simplifies a lot.**
  All callers used to need to decide whether they're a Home- or
  non-Home-style surface to size their bottom inset. Now they just
  call `AtlasBottomModule.height()` — same value everywhere. If a
  future surface needs a different bottom inset (e.g. a sheet that
  covers the module), it should compute its own value rather than
  fork the helper.

- **`snapshot_ui` y-coordinates are the easiest way to verify
  button-position regressions.** Visual diffs from JPEG screenshots
  are too coarse for ~30pt position deltas; the AX frame data is
  exact. Worth running snapshot_ui at every tab/state for layout
  verification, not just relying on screenshot pixel-eyeballing.
