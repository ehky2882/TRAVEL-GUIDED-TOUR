# HANDOFF ‚Äî 2026-07-05 (session 58): bottom-module install race fixed

## What this session did
Fixed the **intermittent "bottom module (mini-player + tab bar) missing on some launches"** bug that the owner and multiple TestFlight testers hit (no reliable repro ‚Äî a timing race). **[PR #357](https://github.com/ehky2882/TRAVEL-GUIDED-TOUR/pull/357) is open, CI + `test_sim` green, and deliberately NOT merged ‚Äî it awaits owner device verification.**

## Root cause (confirmed)
The mini-player + tab bar render in a **secondary `UIWindow`** installed by `Components/BottomModuleWindow.swift` ‚Üí `BottomModuleWindowController.install()`. That method:
- required a `.foregroundActive` `UIWindowScene` and, finding none, **`return`ed silently**; and
- was called **only once**, from the App body's `.onAppear` (`TRAVEL_GUIDED_TOURApp.swift`).

On cold launches where `.onAppear` fired **before** the scene reached `.foregroundActive`, the guard failed, install returned with `window` still nil, and **nothing ever retried it** ‚Üí the module was absent for the whole session. The adjacent `.onChange(of: scenePhase)` only called `dataService.refreshOnForeground()`. The build-71 Home keep-alive change (heavier first layout) plausibly worsened the timing.

## The fix ‚Äî recover instead of giving up once
Files: `TRAVEL GUIDED TOUR/TRAVEL_GUIDED_TOURApp.swift`, `TRAVEL GUIDED TOUR/Components/BottomModuleWindow.swift`, `TRAVEL GUIDED TOURTests/BottomModuleWindowTests.swift` (new).

1. **scenePhase retry (primary).** The App now also calls install on `scenePhase == .active`, factored into a new `installBottomModule()` helper so the `.onAppear` call site and the retry inject the **identical** `@Environment` services ‚Äî the two sites can't drift.
2. **`install()` hardened (belt-and-suspenders).** When no `.foregroundActive` scene exists, it registers a **one-shot `UIScene.didActivateNotification` observer** that retries with the scene that activated (uses `note.object` ‚Üí correct on iPad multi-scene), then removes itself. Now `@discardableResult -> Bool` so a caller knows it deferred.
3. **No double-install.** Both paths still hit `guard window == nil` ‚Äî the window is created exactly once. The install is idempotent, so the repeated `scenePhase == .active` calls are permanent no-ops after the first success.
4. **Color-scheme preserved.** A deferred install re-applies the cached `ColorSchemePreference` via `lastPreference` (the App's `onAppear` `apply()` runs while `window` is still nil, so without this the recovered window would be stuck on SYSTEM appearance).
5. Everything else the window does is untouched: `windowLevel = .normal + 1`, `PassThroughWindow` hit-testing / `interactiveBottomInset`, the `overrideUserInterfaceStyle` `.onChange` sync (no frozen `.preferredColorScheme` reintroduced), toast host, player, deep-link routing.
6. A `Logger` (`OSLog`, subsystem `com.dozent.app`, category `BottomModuleWindow`) logs successful install + a deferral, for owner/device triage.
