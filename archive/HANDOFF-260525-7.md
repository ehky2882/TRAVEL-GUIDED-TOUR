## What shipped this session

### PR #78 — fix(tour-detail): mirror enter-slide to exit by killing hero image crossfade

PR #77 fixed the two big structural fades (inner content insertion
transition; drawer opacity fade-out). What was left: `AsyncImage`'s
default transaction crossfades between phases (`.empty` placeholder
→ `.success` loaded image) over ~250ms. The exit-slide doesn't fire
this because the hero image has been loaded for the entire viewing
session — phase is `.success` the whole time. The enter-slide fires
it: even on a cache hit (the common case here, since `TourListCard`
in the drawer and `PlacecardView` on the map both load the same URL
into URLCache before the user taps), the *first* `AsyncImage` body
eval lands at `.empty` for one frame, then transitions to
`.success` with a 250ms crossfade. That crossfade overlays the
slide and reads as a fade-in stacked on top of the rising layer —
which is why owner said the enter isn't the exact opposite of the
exit even after PR #77.

Fix: new `disableLoadAnimation: Bool = false` parameter on
`HeroImageView`. When `true`, the underlying `AsyncImage` runs with
`Transaction(animation: nil)` — phase changes snap instead of
crossfade. `TourDetailView` opts in for both the single-image hero
and the carousel branch.

Other `HeroImageView` usages keep the default crossfade (drawer
cards, library rows, maker thumbnails, recently-viewed banners) —
those surfaces appear in place, not via a slide-up, so the gentle
crossfade is polish rather than a competing animation.

## If the owner still says enter doesn't mirror

Leads to try next, in order of expected impact:

1. **Pre-render before slide.** Set `displayedTour` on tap; defer
   `presentedTour = tour` by one frame via `DispatchQueue.main.async`
   or `Task.sleep(.milliseconds(16))`. That gives SwiftUI a frame
   to commit the inner content's first body eval and AsyncImage's
   sync cache hit *before* the slide starts. Adds ~16ms latency
   between tap and slide-start, but eliminates any first-render
   side effect bleeding into the slide motion. Trade-off owner may
   or may not accept — start by asking.

2. **Pre-fetch hero image URLs anticipatorily.** When the drawer
   or map renders a tour card / pin, preload the hero image into
   URLCache so any tap on it hits a warm cache. Today this happens
   passively (the card renders the same URL via its own AsyncImage),
   but if the user taps before that card has painted, the cache
   may be cold. Could add a `prefetchHeroImage` call in
   `RecentlyViewedStore.record` or wherever tours enter the user's
   "anticipated" set.

3. **Inspect the bottom-module rectangle fade.** `ContentView` has
   an outer `.animation(.smooth(0.35), value: moduleGeometry)`. The
   conditional `Rectangle` behind the mini-player + tab bar
   (rendered only when `extendsToScreenEdges` is true) fades in
   over 0.35s on enter from Home, fades out over 0.35s on exit
   back to Home. The fade durations are slightly shorter than the
   detail slide (0.4s) so they're not perfectly in lockstep. If
   the owner perceives this as an asymmetry, change either curve to
   match, or use the same animation modifier for both.

4. **Animation curve.** `.smooth(duration: 0.4)` is a critically-
   damped spring with a slow start. The slow start may feel like a
   delay on enter (user just tapped, expects motion) but feel like
   a gentle leave on exit (user dismissed, sheet politely retracts).
   If owner wants snappier enter, try `.spring(response: 0.35,
   dampingFraction: 0.86)` — same general feel with a touch more
   immediate response. Risk: breaks the "exact opposite of exit"
   premise. Discuss before changing.
