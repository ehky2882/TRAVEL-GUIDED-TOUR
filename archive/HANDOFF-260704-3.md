## What shipped ([PR #334](https://github.com/ehky2882/TRAVEL-GUIDED-TOUR/pull/334), squash `34e1c14`)

Both D2 + D3 in one PR (tightly-coupled social-list work on the same RPCs). **No new backend** — every RPC + RLS policy was already live from D1's `backend/social.sql`.

### D2 — Followers / Following lists
- The follower/following **counts** on any maker page (own profile or public creator) are now **tappable** → a list of those profiles; each row pushes that maker's page.
- **`Features/Maker/FollowListView.swift`** (new) — shared `MakerAvatarView` + a compact row (bio, else published-tour count); loading + empty states; rows are `NavigationLink`s to `MakerView`; registers as a pushed detail (full-edge module).
- **`FollowService.followers(of:)` / `following(of:)`** → `list_followers` / `list_following` RPCs. Visibility enforced server-side (a private account's list only returns to its owner).
- **`MakerView`** count pills → `NavigationLink`s (`countLink`) into `FollowListView`.

### D3 — private-account follow requests
- A private account's follows land as `pending`. On the **own profile**, when requests are waiting, a **"N follow requests"** link (`followRequestsButton`) appears under the counts.
- **`Features/Maker/FollowRequestsView.swift`** (new) — requester rows (avatar + name + bio) with **Approve** / **Decline**; actioned rows drop out live; per-row busy guard; empty + loading states. An `onChange` closure refreshes the header count on the profile underneath.
- **`FollowService.pendingRequests()`** → `list_follow_requests` RPC; returns `[FollowRequest]` (the requester's `Maker` + their `user_id`, needed to target the edge). **`approveRequest` / `declineRequest`** are direct table writes — approve flips `follows.status → accepted`, decline deletes — gated by the existing `follows_update_owner` / `follows_delete` RLS.
- New `FollowRequest` model next to `FollowState` in `Data/FollowService.swift`.

### Backend-safety fix
- **`MakerRow.userId` → optional** (`Data/MakerProfileService.swift`). `list_following` can return **seed studios** (null `user_id`), which would otherwise fail the `[MakerRow]` decode and silently empty a following-list. `saveProfile` still always passes a non-nil uid; `asMaker` doesn't use it.
