## Design (`docs/social-design.md`, `backend/social.sql`; commit `23b0ef5`)
Owner decisions (via AskUserQuestion):
- **Follow model** (asymmetric, Instagram-style) — not symmetric "friends".
- **New accounts default PUBLIC** (instant follow; private = approve each request).
- **v1 = counts + lists + approve/decline.** A "tours from creators you follow" home feed is a later extension (no schema change).

One system covers all three notes: public/private + "auto vs manual accept" are the **same switch**; a "friend request" = a pending follow on a private account.

## Backend (owner-applied, hand-held; two SQL blocks, both verified live)
- `makers.is_private boolean default false`.
- `follows(follower_id → auth.users, followee_id → makers, status pending|accepted, created_at)`, PK `(follower_id, followee_id)`.
- `set_follow_status` before-insert trigger: blocks self-follow; sets `status` from the followee's `is_private` (public→accepted, private→pending).
- RLS: insert-as-self; read/delete your own follows + the follows to your maker; followee-owner can approve (update) / decline (delete).
- SECURITY-DEFINER RPCs: `follow_state(m)` → `{followers,following,isFollowing,isPending,pendingRequests}`; `list_followers(m)` / `list_following(m)` (public→anyone, private→owner); `list_follow_requests()`.
- `get_catalog` makers block now returns `isPrivate`.
- **Verified via curl:** catalog carries `isPrivate:false`; `follow_state(<maker>)` returns the 5-key shape.
