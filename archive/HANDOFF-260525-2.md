Three connected fixes bundled into one PR â€” the owner explicitly

**(1) Search no longer hides the bottom module.** `SearchBar` was

- The button row â€” identical layout in both modes (same

  Sits 8pt above the screen bottom in both modes.
- A bottom-extension rectangle below the button row. Transparent on
  Home (8pt â€” the floating-island look, map shows through);
  opaque `secondaryBackground` on every other surface (8pt +
  home-indicator safe-area inset, so the bar bleeds through the

`MiniPlayerBar` didn't need refactoring â€” already had the right

**(3) Detail screens always render with the full-edge module â€” even

### Mechanism notes

- **`AtlasBottomModule.height(extendsToScreenEdges:)`** math updated:
  non-Home is now `MiniPlayerBar.layoutHeight (62) +
  tabBarBackgroundHeight (56) + 8 + safeAreaBottomInset` â€” the extra
  8pt is the new outer gap above the safe-area fill. Was just `62 +
  56 + safeAreaBottomInset` (= 152pt) before; now 160pt. Every
  scrollable non-Home surface reads this helper for its
  `.safeAreaInset(.bottom)` so insets stay in sync automatically.

- **`\.atlasIsHomeTab` env value removed.** No external readers
  outside this repo, and the typed preference is strictly more
  expressive (carries explicit geometry intent, not a tab-name
  proxy). `AtlasModuleGeometryKey`'s `defaultValue` is `.fullEdge` so
  preview / isolated screens render with the safer geometry.

## Process notes / gotchas for next session

- **The handoff's NavigationLink-tap gotcha is real and bit me
  again.** XcodeBuildMCP's `tap` (both `label:` and coordinate
  variants) silently no-ops on most `NavigationLink { destination }
  label: { ... }` sites. My first attempt to verify the search push
  programmatically returned `status: SUCCEEDED` but didn't navigate.
  The same `Continue listening` banner that worked yesterday worked
  again today. There's no obvious pattern for which NavigationLinks
  do or don't take taps â€” some heuristic about label complexity or
  hit-testing in custom backgrounds maybe. **Workaround that did
  work for the search push: tap the back button (or any other
  Button-based affordance) to dismiss the previous push first, then
  fire the same `tap(label:)` on the search bar â€” it took. So the
  flow is taking, just sometimes only on the second attempt within
  a session.** Not chasing further now.

- **SourceKit will fire ~30 spurious `Cannot find 'X' in scope`
  diagnostics during multi-file edits in this project.** Already
  noted in the prior handoff; reconfirmed this session â€” every
  `.swift` file we touched produced a fresh batch of false
  positives. `build_run_sim` was clean on the first run after all
  edits; the test suite passed 84/84 immediately. **Default
  posture: ignore the red squigglies until the next `build_run_sim`
  or `test_sim` run; the SourceKit cache catches up after a real
  build.**

- **`AtlasModuleGeometry` preference is the established way to
  declare bottom-module intent now.** If you add a new pushed
  screen, set `.atlasModuleGeometry(.fullEdge)` on its body
  alongside `safeAreaInset(.bottom, ...)` sized via
  `AtlasBottomModule.height(extendsToScreenEdges: true)`. If you
  ever add a new *tab root*, set whichever geometry that tab wants
  on its outermost view modifier â€” outside the `NavigationStack`'s
  curly braces so pushed children can override.

- **`ContentView` animates the geometry transition with
  `.easeInOut(0.25)`.** Subtle but the right call â€” without it the
  bottom module snaps abruptly between floating-island and full-edge
  mid-nav-transition, which reads as a flicker. Verify visually if
  you change anything in this layout in the future.
