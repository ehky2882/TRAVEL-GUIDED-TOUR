## Root cause (why build 73 didn't fully fix it)
Build 73's [#358](https://github.com/ehky2882/TRAVEL-GUIDED-TOUR/pull/358) cached only the **follow counts**. But **three** things load on the first Me open, and two were still network-gated:
- **`MakerProfileService.myMaker`** (name / bio / avatar / links) — not persisted; starts nil each launch, loads from Supabase on first Me open → placeholder (email local-part) until it returns.
- **`MakerTourService.myTours`** — not persisted; starts empty, loads on first Me open → empty feed until it returns.
- The Me tab (`ProfileView`) is **rebuilt on every tab switch** (`switch`-swapped in `ContentView.tabContent`, unlike the always-mounted Home) and *starts* those fetches lazily on first appearance.
- Knock-on: with `myMaker` empty on first paint, its `avatarURL` is unknown, so even the **build-73 disk-cached avatar** can't render until the row loads.

## Fix — stale-while-revalidate for the whole profile (squash `0670fc1`)
- **New `Data/ProfileSnapshotStore.swift`** — generic, **per-signed-in-user** `Codable` snapshot in `UserDefaults` (keyed by uid; `load/save/clear`). One account never reads another's cache.
- **`MakerProfileService`** — hydrates `myMaker` from the snapshot **synchronously in init** (session is restored synchronously in `AuthService.init`, so the uid is available); `loadMyMaker`/`saveProfile` write through. `hydrateIfUserChanged()` swaps to the right account's snapshot on user change (no-op when unchanged, so it never clobbers a freshly-loaded value).
- **`MakerTourService`** — now takes `auth`; `MakerTour` made `Codable`; hydrates `myTours` in init; `loadMyTours` writes through, keyed by uid.
- **App** — constructs `MakerTourService(auth:)` and **pre-warms** `loadMyMaker`/`loadMyTours` at launch in a non-blocking `Task`, so the refresh is in flight before Me is first opened (not on first tap). `ProfileView.task` calls `hydrateIfUserChanged()` before awaiting the refresh.
- **Result:** first Me paint shows the real profile + tour list instantly (avatar frame-zero too); the network refresh just updates in place.

## Regression guards
- Per-uid keying → no cross-account leak; `hydrateIfUserChanged` is a no-op when the user is unchanged → never overwrites fresher in-memory data with cache.
- `loadMyMaker`/`loadMyTours` still leave state untouched on a network failure (never wipe to placeholder/empty).
- Only the App constructs these two services — the new `auth:` param on `MakerTourService` has no other call sites (grep-checked).
