# HANDOFF — 2026-07-20 (session 63, code, web)

## Shipped: Library-tab jitter fixed → TestFlight 1.1 (21), merged to `main`

**Owner report:** "When I click from home to library (or another tab to library) there's a slight jitter when it loads or re-formats that doesn't look smooth."

**Root cause (reproducible from the code).** The Library tab is **switch-swapped** in `ContentView.tabContent` (rebuilt on every tab entry, unlike Home which stays mounted). So `LibraryView`'s `@State followingMakers` reset to `[]` on each entry, and the **Saved** tab (always the landing section, `selectedSection` defaults to `.saved`) rendered in two steps:
1. Frame 1 — a flat tours-only list (`followingMakers.isEmpty` branch).
2. A `list_following` **network round-trip** later — the layout restructured: a `TOURS` header popped in at the top (shoving tours down) + a `FOLLOWING` section appeared below.

That reflow-on-every-entry is the jitter. The tab-content insertion itself is **not** animated (no `.animation`/`withAnimation` on the switch), so it wasn't a transition-jank issue.

**Fix — [PR #404](https://github.com/ehky2882/TRAVEL-GUIDED-TOUR/pull/404) → `main`, squash `98f2993`** (+307/−32, 4 files). Applied the app's established stale-while-revalidate pattern (already used by `FollowStateStore` for counts, `ProfileSnapshotStore` for the profile, `ImageCache` disk for avatars) to the follow **list**, in two layers:

- **In-memory (kills warm re-entry reflow).** `Data/FollowService.swift` caches the `following(of:)` result in an **observed, per-viewer-scoped** map (stored property on the `@Observable`, so refreshes re-render readers in place). `following(of:)` write-throughs on success (incl. empty = follows nobody → unfollow-to-zero eventually clears) and returns the last-known list on a network error (a blip never reflows a good list away). New `cachedFollowing(of:)` is a **body-safe, viewer-guarded** synchronous read — returns `[]` if the viewer changed since hydration (never another account's graph; the async path re-hydrates via `hydrateFollowingIfViewerChanged()`, called only off-`body`). `followers(of:)`/`following(of:)` public signatures unchanged; refactored the shared fetch into a throwing `fetchMakerList` so success (possibly empty) is distinguishable from a network error for the caching path.
- **Disk (kills the FIRST-load reflow after relaunch).** New **`Data/FollowingListStore.swift`** — a per-signed-in-user, bounded (`limit: 50`) `UserDefaults` blob of `[Maker]` lists keyed by subject maker id, mirroring `FollowStateStore` (`Maker` is `Codable`; per-uid key `followingListCache.<uid>` prevents cross-account leak). `FollowService` **hydrates its in-memory map synchronously at init** for the launch viewer (AuthService restores the session in its own init → uid known). `MakerProfileService.myMaker` likewise hydrates synchronously from its snapshot → on a **cold launch** the Saved tab has both the user's maker id AND their last-known follow list on the first frame.

- **`Features/Library/LibraryView.swift`** — `followingMakers` is now a **computed** reading `followService.cachedFollowing(of: myMaker.id)` synchronously in `body` (was an async-loaded `@State`); `.task(id: authService.userId)` still calls `loadFollowing()` to refresh the cache in place. Net: every Library open — warm re-entry or cold relaunch (after the first launch that seeds the disk blob) — renders the Saved tab's **final layout on the first frame, no re-format**. Only a truly fresh install (nothing cached yet) still fetches once.

- **Behavior-preserving:** same data, same final layout; only the transient empty flash is skipped. Adds `TRAVEL GUIDED TOURTests/FollowingListStoreTests.swift` (round-trip, relaunch persistence, per-account isolation, empty-clears, bounded eviction).

**Verification.** PR `ci.yml` green (iOS Simulator build + Run unit tests incl. the new store tests + validator — the `test_sim` stand-in for a Linux web session; `test_sim` can't run here). **TestFlight 1.1 (21)** built+signed+uploaded via `testflight.yml` `workflow_dispatch` on the branch; **owner device-confirmed the jitter is fixed** (warm switching AND first-load after relaunch) and authorized the merge.

## Build-cut gotcha (codified for next time)

Two `testflight.yml` runs failed **fast (~50s) at the Archive step** with:
```
error: Your account has reached the maximum number of certificates. To create a new one, you must choose a certificate to revoke.
error: No profiles for 'com.ehky.TRAVEL-GUIDED-TOUR' were found … iOS App Development provisioning profiles …
```
This is **not a code error** (CI was already green). The cloud-Mac signing had accumulated **Apple Development** certificates to Apple's cap. Key points:
- The error names **iOS App Development** — the archive does a *development* signing pass, and it's the **Apple Development** cert type at the cap. Revoking **Distribution** certs does **not** help.
- Revocation has a **few-minute propagation** delay in Apple's signing service.
- Owner revoked the Apple Development-type certs → the rebuild signed clean.
- **Recurs** because each cloud archive mints a fresh dev cert. Durable follow-up **offered, owner deferred:** switch `testflight.yml`'s archive to **distribution-only signing** so it stops creating per-build dev certs.
- **Runbook on this exact fast-fail:** revoke the **Apple Development** certs specifically at developer.apple.com → Certificates, IDs & Profiles → Certificates, wait ~5 min, rebuild.
