## What shipped this session

### PR #77 — fix(tour-detail): clean slide via synchronous content + drawer z-swap

Resolves the open animation issue tracked in
`docs/open-issues/tour-detail-slide-animation.md`. Two changes, both
in `ContentView.swift` + `Features/Tour/TourPresenter.swift`. No
behavior changes outside the slide animation path.

**Root cause #1 — inner content fade-in.** `displayedTour` lived on
`ContentView` as `@State` and was mirrored from
`tourPresenter.presentedTour` via `.onChange`. `.onChange` fires one
render tick AFTER the layer's offset animation has already started.
So the `if let displayedTour` conditional inserted the
`NavigationStack { TourDetailView }` mid-slide, and SwiftUI filled
that view-tree gap with its default opacity-fade transition. The
user saw a colored rectangle sliding up + content fading in on top
of it = "fade-in" perception.

**Fix:** `displayedTour` moved onto `TourPresenter`, updated
synchronously inside `present(_:)`. Now the inner content is in the
view tree on the SAME SwiftUI tick the offset starts animating —
no one-tick gap, no SwiftUI default-transition fill. `dismiss()`
still keeps the lag (cleared 0.45s later via a `Task` sleep) so the
content stays rendered while the layer slides off-screen.
`.transition(.identity)` on the inner content as belt-and-suspenders
against any future SwiftUI default-transition behavior.

**Root cause #2 — drawer opacity-fade asymmetry.** PR #76 wrote the
drawer with `.opacity(presentedTour == nil ? 1 : 0)` + `.animation(.smooth(0.4))`. The drawer was z-stacked ABOVE the
detail layer, so its opacity-fade dominated the visible motion. From
the drawer entry point (drawer at `.large` detent, covering most of
the screen), the fade-out was the whole story. From the placecard
entry point (drawer at `.peek` detent, only 80pt at bottom), only
the bottom strip faded so the slide stayed visible. Same code path,
different visual — the exact asymmetry the owner reported.

**Fix:** drawer no longer animates opacity. Its `.zIndex` swaps
based on `displayedTour`:

| State | drawer zIndex | rationale |
|-------|---------------|-----------|
| no detail | 4 | above mini-player + tab bar — PR #76's "last card visible at scroll-end" fix preserved |
| detail active | 1 | below the detail layer (z-2) so the slide-up covers and the slide-down reveals it naturally |

Mini-player + tab bar stay pinned at z-3, always above the detail
layer (z-2). The action bar's `secondaryBackground` color extends
down through the bottom-module area so the seam is invisible. Tab
bar + mini-player buttons remain tappable through the entire detail
session. Drawer hit-testing is gated by `displayedTour == nil` so
taps that miss the X button can't sneak through to drawer cards
underneath.
