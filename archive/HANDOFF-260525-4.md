## What shipped this session

### PR #70 — buttons identical across surfaces; only background fill changes (merged)

Commit `643cbd7` on `main`. 3 files, +67 / −90.

Owner's review of PR #69's sim build:

> *"the buttons themselves between the home 'island' state is different from the other (edge to edge) tabs. they should always be the same. only different is the fill to edge! conform to 'home' position."*

PR #69 anchored the buttons at the right *y-position* (matching OLD Home) but still varied the bar's HORIZONTAL geometry between modes — 8pt inset + rounded bottom corners on Home, 0 inset + square corners on non-Home. That made the buttons land at different x / width values per surface (x=8 w=128.67 on Home vs. x=0 w=134 on Library/Me). Owner's design rule: buttons identical everywhere; ONLY the edge fill differs.

Implementation:

- **`AtlasTabBar` always renders the same inset rounded pill.** Dropped the `extendsToScreenEdges` flag entirely. Single form: 8pt horizontal inset, phone-screen-radius rounded bottom corners, transparent 8pt strip below. The view doesn't know or care which tab it's on.
- **`MiniPlayerBar` always renders at 8pt `sideInset`.** Dropped the conditional. Single form.
- **`ContentView` paints an edge-to-edge `secondaryBackground` rectangle BEHIND the inset VStack** on non-Home surfaces (or when `navState.isShowingDetail`). Same color as the bar, so the 8pt side + bottom gaps blend into the fill and the bottom region reads as a continuous full-width strip. On Home root no fill is painted — gaps show the map.

The "floating-island vs full-edge" distinction is now a SINGLE concern (a fill-or-not decision) handled in one place. Adding new pushed screens doesn't need to think about geometry — `navState.push()` / `pop()` engages/disengages the fill.

Final `snapshot_ui` anchor:

| Surface  | Home button     | Library button     | Me button         |
|----------|-----------------|--------------------|-------------------|
| Home tab | x=8, w=128.67   | x=136.67, w=128.67 | x=265.33, w=128.67 |
| Library  | x=8, w=128.67   | x=136.67, w=128.67 | x=265.33, w=128.67 |

Identical, and matching OLD Home exactly. Mini-player elements ("Hello! Ready to explore?" text at x=32 y=756 w=242, skip-forward at x=282 y=750, play at x=334 y=750) likewise identical between tabs.

Owner approval: *"this is what i want."*

## Process notes / gotchas for next session

- **The "buttons identical, fill behind differs" design rule is now in memory** as `feedback-atlas-module-design.md`. If anyone proposes a conditional layout inside `AtlasTabBar` or `MiniPlayerBar` keyed off a mode flag, ASK before merging — owner has rejected that pattern twice. The pattern that works: single-form bar in a ZStack with a separately-rendered fill rectangle behind it.

- **`AtlasNavigationState` Observable is still load-bearing.** Pushed views (`TourDetailView`, `MakerView`, `ManageDownloadsView`, `SearchView`) call `navState.push()` on `.onAppear` and `navState.pop()` on `.onDisappear`. ContentView reads `navState.isShowingDetail` for the fill-or-not decision. Don't remove these calls — they're what makes "Home → Detail engages the fill behind the bar" work.

- **`AtlasBottomModule.height(extendsToScreenEdges:)` parameter is a no-op.** Returns 126pt regardless. Kept for self-documenting call sites. Safe to drop entirely whenever someone cleans it up.

- **`xcodebuild archive` path.** The runbook in `docs/testflight.md` still references `~/Desktop/"TRAVEL GUIDED TOUR"`. The repo actually lives at `/Users/EY/TRAVEL-GUIDED-TOUR`. Worth updating the doc next time someone touches it.

- **TestFlight 1.0 (9) is a "before/after" build for the bottom module.** Owner will compare it to what they saw in sim. If anything reads differently on device than in sim — particularly the cluster pin sizes or the floating-island gap — surface it next session before the next build bump.
