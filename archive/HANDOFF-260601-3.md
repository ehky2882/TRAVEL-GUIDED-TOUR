#### Drawer-gap bug found mid-review (worth remembering)

After PR #104's initial push, owner asked to make the drawer max
detent exactly 8 pt below the chip row. Math said it already was
(both gaps = `AtlasSpacing.sm`). Diagnostic colors (magenta on
the search/chips block) showed the **actual** gap was ~67 pt — a
real bug, not perception.

Root cause: `BottomSheet.heightForDetent(.large)` was computing
`topGap = topInset + topReservedHeight`. The GeometryReader's
bounds already start below the device top safe area (ContentView's
ZStack respects top safe area), but SwiftUI's
`geo.safeAreaInsets.top` still **reports** the device's actual
inset value — it describes the device, not what's left to
consume. So the `+ topInset` was double-counting.

Fix: removed `topInset` from the `.large` formula in `BottomSheet`
and from both `drawerVisibleHeight` mirrors (`HomeView`,
`HomeDrawerContent`). Gap is now mathematically and visually
8 pt.

Pairs naturally with `feedback-visual-debugging.md` — bright
contrasting diagnostic colors saved the day again. Worth
remembering on any future sheet/window geometry bug.
