## How it was built (capability note)

**First time the on-demand TestFlight CI was used from a web session purely to
produce a build for OWNER REVIEW (not a milestone ship).** No Mac, no local
Xcode. Path:
1. `Skill`/tools: pushed the branch, then triggered
   **`.github/workflows/testflight.yml`** via `actions_run_trigger`
   (`run_workflow`, `ref = the branch`) — the `workflow_dispatch` entry point.
2. Run [29687284651](https://github.com/ehky2882/TRAVEL-GUIDED-TOUR/actions/runs/29687284651):
   Archive (build+sign) + Export/upload both green in ~4 min →
   **TestFlight 1.1 (9)** (build number = `github.run_number`).
3. Owner device-reviewed → "Merge it."
4. Opened PR #398, waited for its CI (Validate + iOS Simulator build +
   **Run unit tests** — the unit-test job is the substitute for `test_sim`,
   which can't run on Linux; it confirms the test target still compiles after
   the saved-makers test removal), squash-merged on green.

**Gotchas hit:**
- `gh` CLI is **not installed** in the web session — a `Monitor` script using
  `gh api` silently never detected completion (looped on `api-error`). Use the
  **`mcp__github__*`** tools (or `curl` to `api.github.com`, public repo) to poll
  run/job status, not `gh`.
- `mcp__github__actions_list` `list_workflow_runs` output is **huge** (>token
  cap) — it spills to a file; slice with `python3 read()[A:B]` or just use
  `get_workflow_run` by id.

## Owed / follow-ups

- **Delete the merged branch** `claude/maker-follow-bookmark-redundancy-e65u9d`
  **in the GitHub UI** — this repo's git proxy blocks `git push --delete`
  (documented limitation).
- **Supabase `user_saved_makers` table** is now unused but NOT dropped
  (owner-run infra; harmless to leave). Drop via SQL Editor whenever tidying:
  `drop table if exists public.user_saved_makers;`
- **Library "Following" caveat (by design):** it only populates once the user's
  own `makers` row is loaded — identical to how the profile's own Following list
  already behaves. A signed-in consumer who has never created a maker profile
  sees tours-only in Saved. If that ever feels wrong, the fix is a
  `list_my_following` RPC keyed by `auth.uid()` (no maker id needed) rather than
  `list_following(m)`.
