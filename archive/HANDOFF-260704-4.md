# HANDOFF 2026-07-04 (session 56) — perf: keep Home + map alive across tab switches (fixes return-to-Home lag)

Code session. Fixed a felt performance issue the owner reported on device: **returning to the Home tab from Library/Me had a noticeable lag/load.** One focused PR ([PR #347](https://github.com/ehky2882/TRAVEL-GUIDED-TOUR/pull/347), squash `b3441c6`), owner-reviewed on device then authorized merge ("merge it").

## Root cause (profiled, not guessed)
`ContentView.tabContent` rendered the three tab roots with a `switch appShared.selectedTab`, so **leaving Home unmounted `HomeView` entirely**; returning **rebuilt it from scratch** — including the SwiftUI `Map`, which reloads tiles and re-adds/re-clusters the full **~509-stop** annotation set on construction, and lost `HomeView`'s `@State` camera position back to the NYC default.

**Correction to the original brief's diagnosis:** the map is SwiftUI's **native `Map`**, NOT a hand-wrapped `UIViewRepresentable MKMapView`. The thesis (switch → unmount → costly rebuild) still held.

**Profiling (temporary `CFAbsoluteTimeGetCurrent` logs in `clusterItems` + `filteredTours`, reverted before commit):** the Swift-side recompute is **negligible** — `filteredTours` ≈**0.02 ms**, cluster of 509 markers ≈**1–3 ms**, and the map stops recomputing once idle. So the lag is the map's **UIKit reconstruction on remount**, which staying mounted eliminates. Per the brief's own conditional (memoize only if `filteredTours`/cluster is a measurable contributor), **no memoization was added** — it would add risk for no measurable gain.

## The fix (`ContentView.swift` + `Features/Home/HomeView.swift`, 2 files, ~40 lines)
- `tabContent` → a **`ZStack`** that keeps **`HomeView` permanently mounted** (visibility toggled by `selectedTab` via `.opacity` / `.allowsHitTesting` / `.accessibilityHidden`), with **Library/Me overlaid opaque** on top when active (they keep their existing mount-on-select lifecycle). Returning to Home is a **re-show, not a rebuild**; the map camera/zoom is preserved.
- **Off-screen Home is inert:** `allowsHitTesting(false)` blocks all map gestures (so the 60 fps `.continuous` camera callbacks can't fire), and a new **`isActive: Bool`** param on `HomeView` short-circuits `onCameraChanged` / `onCameraMoving` so a late *programmatic* map settle frame (e.g. a recenter animation finishing just after a tab switch) can't retract the drawer or mutate shared map state off-screen.

## Regression guard
Untouched: drawer detent persistence (`homeSheetDetent`), `HomeSharedState` map↔drawer wiring, the tour-detail slide-up (`TourPresenter`) + maker (`MakerPresenter`) UIKit layers, deep-link routing, and the `onChange(selectedTab)` side effects. `LocationManager` streams heading/location continuously regardless of view lifecycle, so keeping Home mounted doesn't change it (verified via an Explore pass over `AtlasNavigationState`/`SearchView`/`BottomModuleRoot`/`LibraryView`/`ProfileView`/`LocationManager`).
