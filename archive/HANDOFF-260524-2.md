### Mini-player end-of-tour state — PR #61, merged

Commit `c054a67` on `main` (squash-merged). 2 files, +68 / −17.

**Problem:** when a tour finished, the mini-player briefly flashed
through `"Loading…"` + the hourglass glyph, and tapping play opened
the full PlayerView instead of restarting in place.

**Root causes** (two layered):

1. `AVQueuePlayer` drains its queue at `didPlayToEndTime`. A
   parameterless `play()` had nothing to resume — only the full
   player's stop/intro path could restart audio. So the mini-player's
   `.ended` case fell back to `onExpand()`.
2. AVPlayer's `timeControlStatus` emits a transient
   `.waitingToPlayAtSpecifiedRate` after end-of-item. The previous
   observer only protected `.failed` from being clobbered to
   `.loading`; `.ended` (and `.idle`) were vulnerable, so the bar
   re-entered the loading visual moments after the tour finished.

**Fix** (`AudioPlayerService.swift` + `MiniPlayerBar.swift`):

- `AudioPlayerService` now retains `lastPlayedURL` and exposes
  `replayLast()`, which re-issues the original `play(url:...)` so the
  mini-player can restart in place.
- `observePlayerStatus` preserves `.failed`, `.ended`, and `.idle`
  through `.waitingToPlayAtSpecifiedRate` — only `.loading`,
  `.playing`, `.paused` may be downgraded to `.loading`.
- `MiniPlayerBar`: progress ring forced to 1.0 at `.ended` (so the
  arc reads as a complete circle even when AVPlayer stops reporting
  time a touch shy of `duration`). `.loading` now falls through to
  the maker-name + play-glyph visual — no more flicker. Tap-on-`.ended`
  calls `replayLast()`. Voice-over reads "Replay tour" on `.ended`.

**Verification path** (used for sim review, passes manually):
play any tour → open full player → scrub to ~3s before end → let
finish → back to home → mini-player shows full ring + author name +
play icon, tap restarts from 0:00 without opening the full player.

**Pre-existing Swift 6 warning** carried forward:
`AudioPlayerService.swift` line 369 (was 344 pre-fix) —
`main actor-isolated conformance of 'PlaybackState' to 'Equatable'
cannot be used in nonisolated context`. Pre-existed at the
`state == .failed` comparison in the periodic time observer; the
fix does not touch that line, the warning just shifted down due to
added lines above. Project is on Swift 5.0 → future-error, not a
blocker.

## Tribal knowledge from this session

- **Process error worth remembering**: I built the simulator from
  the wrong branch first (working tree had drifted back to
  `claude/home-visual-polish-bundle` while the fix lived on the
  newly-cut `claude/miniplayer-end-of-tour-fix`). The owner caught
  it because the symptoms they reported on the sim ("Loading…" +
  hourglass at end of tour) were *exactly* the strings my fix had
  removed from the source — that mismatch is the diagnostic
  fingerprint of "you built the wrong tree." Verify `git branch
  --show-current` immediately before any `build_run_sim` whenever
  the same working tree has been used across multiple branches in
  one session.
- **Parallel sessions on the same repo** are now a live
  possibility — the polish branch's `5db7aaa` was made by a
  concurrent Claude session while this session was active. Reflog
  is the clearest tell (`reset: moving to origin/main` + a new
  commit hash without you having done either). Don't push or
  rebase another session's branch.
- **`AVQueuePlayer` semantics**: at `didPlayToEndTime` the queue
  drains. `play()` (parameterless) is a no-op until a new item is
  inserted. Any "replay" affordance must re-issue `play(url:...)`,
  not just call `play()`. The new `replayLast()` on
  `AudioPlayerService` exists for exactly this — reuse it rather
  than re-deriving the URL in the view layer.
- **`waitingToPlayAtSpecifiedRate` is sticky after end-of-item.**
  AVPlayer keeps emitting it; any state machine that maps it to
  `.loading` *must* gate on the previous state, not blindly
  transition. The audit-style switch in `observePlayerStatus`
  enumerates exactly which previous states allow the downgrade.
