## What shipped this session

### PR #76 — tour detail as slide-up layer + drawer hoist + chrome unification

One bundled PR covering five connected changes that together replace
the old NavigationLink-push presentation of `TourDetailView` with a
custom slide-up layer hosted at the `ContentView` level. The bundle
is large because the changes are mutually-dependent (lifting the
drawer requires shared-state plumbing, which requires the layer-host
ContentView already exists, which requires the presenter, etc.).

1. **Home drawer hoisted out of `HomeView` into `ContentView`.**
   - New `Features/Home/HomeSharedState.swift` (`@Observable`) carries
     the state the map and the drawer both touch: `selectedCategory`,
     `placecardTour`, `placecardCoordinate`, `visibleRegion`,
     `sheetDragOffset`. Injected via `@Environment`.
   - New `Features/Home/HomeDrawerContent.swift` extracts the drawer
     body (header, scroll list, quick-resume banners, empty state).
     The drawer's tour-tap is now a closure parameter (`onTourTap`)
     so the host decides presentation policy.
   - `HomeView` keeps the map + search bar + chips + map control
     stack; lost the drawer, the placecard-driven `navTargetTour`
     and its `.navigationDestination`. Map state (`selectedCategory`,
     etc.) reads from the shared state via `@Bindable`.
   - `ContentView` hosts a `BottomSheet` containing `HomeDrawerContent`
     z-stacked AFTER the mini-player + tab bar so the drawer sits ON
     TOP of them — fixes the long-running "last card peeks behind
     the tab bar" + "can't reach scroll-end" complaint that survived
     PR #69's `clipped()` attempt.
   - The drawer is now ALWAYS rendered when `selectedTab == .home`
     (previously conditional on `presentedTour == nil`), with
     `.opacity` + `.allowsHitTesting` driving its visibility. Toggling
     the drawer's mounted/unmounted state on the same SwiftUI tick
     as the detail-layer transition was conflicting with the layer's
     animation.

2. **Tour detail always presented as a slide-up layer.**
   - New `Features/Tour/TourPresenter.swift` (`@Observable`): one
     property `presentedTour: Tour?`, two methods `present(_:)` and
     `dismiss()`.
   - `ContentView` adds a `displayedTour` `@State` that lags
     `presentedTour` on dismiss (cleared via a `Task.sleep(0.45)`)
     so the detail content stays rendered while the layer slides
     off-screen.
   - The layer is ALWAYS in the view tree, position controlled via
     `.offset(y: presentedTour == nil ? screenHeight : 0)`. Off-screen
     offset uses the **actual screen height** (looked up via
     `UIWindowScene`) so the slide is fully visible across its full
     0.4s duration — early attempts used `2000` which left ~57% of
     the animation off-screen and the visible tail read as a fade.
   - Entry-point conversions: `TourListCard` (NavigationLink →
     button → presenter), `RailCarousel`, `LibraryView`, `SearchView`
     result rows, `HomeDrawerContent` quick-resume banners,
     `HomeView`'s placecard. `MakerView`'s in-stack push KEEPS its
     NavigationLink because the layer wraps `TourDetailView` in its
     own `NavigationStack` and MakerView pushes onto THAT stack.

3. **`TourDetailView` X close.**
   - `.navigationBarBackButtonHidden(true)` + X icon in
     `.topBarLeading` toolbar slot calls `tourPresenter.dismiss()`.
   - `.toolbarBackground(AtlasColors.secondaryBackground, for:
     .navigationBar)` + `.toolbarBackground(.visible)` so the nav bar
     matches the rest of the detail surface — without this, the nav
     bar showed a system background that contrasted against the
     `secondaryBackground` scroll view below it.

4. **Mini-player + tab bar stay visible underneath the detail layer.**
   - `moduleGeometry` in `ContentView` now reads
     `tourPresenter.presentedTour != nil` (alongside
     `navState.isShowingDetail`) so the module switches to
     `.fullEdge` on the SAME SwiftUI tick the layer slides up.
   - `.animation(.smooth(0.35), value: moduleGeometry)` replaces the
     old `.easeInOut(0.2)` so the module morphs on the same curve
     family as the layer's slide.

5. **SearchBar + chips background.**
   - `SearchBar.swift`: `.regularMaterial` capsule + 1pt stroke →
     `AtlasColors.secondaryBackground` capsule, no stroke.
   - `CategoryChipRow.swift`: unselected chips were
     `.regularMaterial` + stroke → `AtlasColors.secondaryBackground`,
     no stroke. Selected chip unchanged (`primaryText` fill).
   - One unified chrome color across drawer / mini-player / tab bar /
     search bar / chips.

**Z-order in `ContentView`'s outer ZStack:** `tabContent` → tour
detail layer (always rendered, offset-controlled) → mini-player +
tab bar (always on top) → home drawer (when home tab,
opacity-controlled). The detail layer is BELOW the bottom module so
the mini-player + tab bar remain visible above it; the drawer is
ABOVE the bottom module so its bottom edge sits flush against the
mini-player's top without peeking behind.

## ⚠️ Open issue — tour-detail slide animation (PR #77, follow-up)

**Owner-flagged, multiple fix attempts unsuccessful in this session,
explicitly tracked in PR #77.** Quoting the owner's final feedback:

> *"the animation of sliding up and x close (slide down) needs a lot
> of refinement. very clunky."*
>
> *"no good. it should feel like a full sheet (edge to edge) coming
> up and full sheet going down"*
>
> *"full screen done but the slide animation still not feeling clean.
> on exit, still fading out."*
>
> *"from the drawer - there is hardly animation. fade in, fade out.
> from the placecard, fade in, slide down out. so not even the same."*
>
> *"this is not working. issue is not getting fixed. make note of
> this issue in the main and issue a new pr just for this."*

Fix attempts logged in this session, none satisfied the owner:

| # | Approach | Outcome |
|---|----------|---------|
| 1 | `.sheet(item:)` (system modal) | Covers mini-player + tab bar — rejected (owner wants those visible). |
| 2 | Custom layer in ZStack with `.transition(.move(edge: .bottom))` + `.clipShape` + `.shadow` + `withAnimation(.smooth)` | Slide-up works; dismiss reads as fade. |
| 3 | Strip `clipShape` + `shadow`, add background extending top safe area | Slide-up covers full screen (no map gap at top); dismiss still fades. |
| 4 | ZStack-wrap layer to give it single SwiftUI identity, always-render drawer with opacity | Dismiss still fades from owner's perspective. |
| 5 | Switch from `.transition` to explicit `.offset` + `.animation(_, value:)` with lagging `displayedTour` | Owner: drawer-case → fade both ways; placecard-case → fade in / slide-down out. **Different behavior per entry point.** |
| 6 | Cap off-screen offset to actual screen height (was 2000), animate drawer opacity on the same 0.4s clock | Still not satisfactory per owner. Session ended here. |

Current code uses approach 6 (offset-based, lagging displayedTour,
actual screen height for off-screen offset, drawer opacity animated
with `.smooth(0.4)`). PR #77 owns the fix.

Best leads for PR #77:
- **Capture the actual rendered animation** (video / Instruments
  Time Profiler) — owner's reported "fade vs slide" distinction
  between drawer-case and placecard-case suggests the offset IS
  animating but some other view-tree change (drawer mount/unmount,
  the displayedTour lag) is masking it.
- **Move state ownership.** Both `presentedTour` AND `displayedTour`
  should live on `TourPresenter` so they update synchronously inside
  `present(_:)` / `dismiss()`. Right now `displayedTour` is on
  `ContentView` and updated via `.onChange`, which fires AFTER the
  offset animation starts — the inner `if let tour = displayedTour`
  conditional flickers content in/out during the slide.
- **Drawer-case-specific.** When opened from the drawer, the drawer
  was the dominant visible element pre-tap. Its `.opacity` change
  alongside the detail's offset may overshadow the slide entirely.
  Consider holding the drawer at `opacity 1` UNTIL the detail is
  fully up (delayed fade) so the slide is the dominant motion.
- **Nuclear option:** `UIViewControllerRepresentable` bridging a
  custom UIKit presentation that mimics the system sheet but exposes
  a configurable bottom inset (so the mini-player + tab bar remain
  visible). Loses the SwiftUI-idiomatic feel but gets pixel-perfect
  iOS sheet motion.
